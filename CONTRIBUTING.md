# Contributing to Azure Infrastructure Using Bicep

Thank you for considering contributing to the Azure Infrastructure Using Bicep project! We appreciate your contributions and want to ensure a smooth process for everyone involved. Below are the guidelines we ask contributors to follow.

## Code Standards
- Follow the official Bicep guidelines for syntax and formatting.
- Use meaningful variable names to enhance code readability.
- Consistently format your code and maintain proper indentation.
- Include comments where necessary to explain complex logic.

## Commit Conventions
- Use clear and descriptive commit messages.
- Follow the format: `type(scope): subject`
  - Type can be one of the following: `feat` (new feature), `fix` (bug fix), `docs` (documentation), `style` (formatting, missing semicolons, etc.), `refactor` (refactoring code), `test` (adding tests), `chore` (maintenance tasks).
- Example: `feat(validation): add validation to input parameters`

## Pull Request Process
1. **Fork the repository** and clone it to your local machine.
2. **Create a new branch** for your feature or fix:
   ```
   git checkout -b feature/my-feature
   ```
3. **Make your changes** and ensure everything works correctly.
4. **Commit your changes** following the conventions above.
5. **Push to your fork**:
   ```
   git push origin feature/my-feature
   ```
6. **Open a pull request** on the main repository and describe your changes, referencing any related issues.

By following these guidelines, you'll help maintain a clean and efficient workflow for the project. Thank you for your contributions!
